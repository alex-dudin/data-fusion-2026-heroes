# LNS Solver для решения задачи Heroes VRPTW

LNS солвер есть в виде Python-версии и в виде C++ версии. Они абсолютно идентичны: у них одинаковый алгоритм, одинаковые CLI-опции, даже комментарии одинаковы. Единственное отличие в скорости: С++ вариант в 50-100 раз быстрее.

Далее достаточно кратко описан алгоритм работы LNS, более подробно реализацию алгоритма можно изучить по исходникам, в них очень много комментариев. Также рекомендую воспользоваться визуализатором алгоритма (`lns_demo.py`), он реализует тот же набор операторов и подходов, как и солвер.

## Что делает LNS Solver

1. загружает данные о героях, мельницах и расстояниях;
2. **разбивает задачу** на 7 отдельных дней;
3. для каждого дня строит маршруты героев;
4. улучшает маршруты с помощью **LNS**:
   - часть решения удаляется (фаза `destroy`),
   - потом перестраивается (фаза `repair`);
5. объединяет маршруты всех дней в один итоговый `submission.csv`.

## Как работает алгоритм

**LNS (Поиск в большой окрестности)** — это метаэвристический алгоритм, предложенный **Полом Шоу (Paul Shaw)** в 1998 году.

> Shaw, P. (1998). *Using constraint programming and local search methods to solve vehicle routing problems.* Proceedings of CP-98, LNCS 1520, pp. 417–431.

В отличие от классического локального поиска, который исследует **маленькие окрестности** (например, замена одного элемента), LNS работает с **большими окрестностями** — на каждой итерации значительная часть решения разрушается и строится заново.

### Идея очень простая

1. Сначала строим начальное решение жадным способом.
2. Потом много раз:
   - ломаем часть текущего решения;
   - пытаемся собрать его заново получше;
   - если новое решение лучше — принимаем;
   - если хуже — иногда всё равно принимаем, чтобы не застрять в плохом локальном варианте (simulated annealing).

### Преимущества и недостатки

#### Преимущества
- **Эффективно избегает локальных оптимумов** за счёт больших изменений
- **Интуитивно понятен** — логика «разрушить и построить заново»
- **Гибкость** — легко адаптировать к конкретной задаче
- **Высокое качество** решений на практике (SOTA)

#### Недостатки
- Нужно **проектировать операторы** под конкретную задачу
- Качество сильно зависит от **выбора операторов**
- Параметр *q* (степень разрушения) требует **настройки**
- Итерации **дороже**, чем в простом локальном поиске

### Операторы разрушения

Используется всего два оператора разрушения:
- `random` — удаляем случайные мельницы;
- `worst` — удаляем самые "неудобные" мельницы.

Оператор `worst` работает так:
1. сортируем кандидатов по выгоде удаления;
2. берём верхние `rcl_size` кандидатов;
3. случайно выбираем одного из них.

### Операторы восстановления

Используется два оператора восстановления:
- `greedy` — вставляем мельницу туда, где это дешевле всего;
- `regret2` — раньше вставляем "хрупкие" мельницы, которые потом может быть трудно вставить.

#### Основная идея оператора regret2

Не все мельницы одинаково "срочные".

Для одной мельницы может быть:
- один очень хороший вариант вставки,
- и второй вариант сильно хуже.

Если такую мельницу не вставить сейчас,
позже хороший вариант может пропасть,
и мельницу станет трудно или невозможно вставить.

Поэтому для каждой мельницы считаем:
- `best1` = лучшая стоимость вставки;
- `best2` = вторая лучшая стоимость вставки;
- `regret = best2 - best1`.

Чем больше `regret`, тем важнее вставить мельницу прямо сейчас.

На каждом шаге выбираем мельницу с максимальным `regret`.

## Важное упрощение модели

Этот solver использует **посуточную декомпозицию**:

- день 1 решается отдельно;
- день 2 отдельно;
- ...
- день 7 отдельно.

Между днями переносится только упрощённое состояние героя:
- где он закончил предыдущий день;
- сколько условно осталось движения.

Данный подход резко снижает размерность решаемой задачи за счет потенциального снижения эффективности решения. На самом деле, так как максимальное число мельниц открывается в первый день, то такой подход практически не влияет на эффективность решения. 

## Как собрать C++ версию

Нужен компилятор с поддержкой **C++20**.

Никаких зависимостей нет, используется только стандартная библиотека C++.

### Подойдут компиляторы:

- **g++ 13+**
- **clang++ 16+**
- **MSVC / Visual Studio 2022** с поддержкой C++20

### Linux / macOS / MinGW

```bash
g++ -O2 -std=c++20 lns_solver.cpp -o lns_solver
```

Если хотите более быструю сборку:

```bash
g++ -O3 -std=c++20 -march=native -fomit-frame-pointer -funroll-loops lns_solver.cpp -o lns_solver
```

### Windows (MSVC)

Откройте **Developer Command Prompt for Visual Studio** и выполните:

```bat
cl /O2 /std:c++20 lns_solver.cpp
```

## Как запустить

Пример запуска Python-версии:

```bash
python lns_solver.py \
  --data-dir data \
  --output-dir out \
  --heroes 20 \
  --day-time-limits 60 \
  --seed 42 \
  --iterations 0 \
  --rcl-size 5 \
  --destroy-frac-min 0.10 \
  --destroy-frac-max 0.35 \
  --temp-start 0.20 \
  --temp-end 0.001 \
  --log-every 100
```

Пример запуска C++ версии:
```bash
./lns_solver \
  --data-dir data \
  --output-dir out \
  --heroes 20 \
  --day-time-limits 60 \
  --seed 42 \
  --iterations 0 \
  --rcl-size 5 \
  --destroy-frac-min 0.10 \
  --destroy-frac-max 0.35 \
  --temp-start 0.20 \
  --temp-end 0.001 \
  --log-every 100
```

## Основные параметры

### Пути
- `--data-dir` — путь к папке с входными данными
- `--output-dir` — путь к папке, куда сохранить результат

### Основные настройки
- `--heroes` — сколько первых героев использовать
- `--day-time-limits` — лимит времени на каждый день
- `--seed` — random seed
- `--iterations` — максимум итераций LNS на день  
  - `0` означает: работать до исчерпания лимита времени
- `--log-every` — как часто печатать лог

### Настройки LNS
- `--rcl-size` — размер списка лучших кандидатов в `destroy_worst`
- `--destroy-frac-min` — минимальная доля удаляемых мельниц
- `--destroy-frac-max` — максимальная доля удаляемых мельниц
- `--temp-start` — начальная температура simulated annealing
- `--temp-end` — конечная температура

### Параметр `--day-time-limits`

Можно задать:

#### одно число для всех дней
```bash
--day-time-limits 60
```

Тогда на каждый день будет по 60 секунд.

#### или 7 чисел отдельно
```bash
--day-time-limits 20,20,30,30,40,40,50
```

Тогда:
- день 1 — 20 секунд
- день 2 — 20 секунд
- день 3 — 30 секунд
- ...
- день 7 — 50 секунд

## Что сохраняется

После работы скрипт создаёт в `output-dir`:

### `submission.csv`

Файл с маршрутом в формате:

```csv
hero_id,object_id
1,10
1,25
2,7
...
```

Именно этот файл можно потом проверять визуализатором или оценщиком.

### `summary.txt`

Краткая статистика запуска:
- сколько мельниц собрано;
- сколько потрачено движения;
- сколько осталось движения;
- награда;
- стоимость героев;
- итоговый счёт.

## Как можно улучшить существующий LNS-солвер

### 1. Добавить более сильные destroy-операторы

Сейчас есть:
- random;
- worst.

Это хороший старт, но для LNS почти всегда очень помогает **ALNS-style набор операторов**.

#### Shaw removal / related removal

Удалять похожие объекты:
- близкие по расстоянию;
- часто встречающиеся рядом в маршрутах.

Это хорошо разрушает локальные кластеры.

#### Route removal

Удалять сразу кусок маршрута одного героя.

### 2. Добавить более сильные repair-операторы

Сейчас есть:
- greedy,
- regret2.

#### Candidate list
Для объекта искать вставку не во все маршруты и не во все позиции, а:
- в ближайшие подходящие маршруты;
- в небольшое число лучших позиций.

#### Beam search repair

Не один greedy-вариант восстановления, а хранить `B` лучших частичных восстановлений.

#### Exact mini-repair

Если после destroy осталось, например, 15–30 объектов, то этот кусок можно пересобрать MIP/CP-SAT-ом **точно**.

### 3. Добавить локальный поиск после repair

Сейчас восстановил — и пошел дальше.  
Очень полезно после repair делать небольшую локальную полировку.

#### Внутри маршрута героя

- алгоритм 2-opt;
- алгоритм or-opt;
- relocate одного объекта;
- swap двух объектов.

#### Между героями

- relocate объекта между героями;
- swap объектов между двумя героями;
- exchange tails.

### 4. Перейти к ALNS

ALNS (Adaptive Large Neighborhood Search) является расширением метода LNS.

#### Что делает его «адаптивным»?

Алгоритм **автоматически выбирает**, какие операторы использовать, на основе их прошлой эффективности. Каждому оператору присваивается **вес (score)**, который обновляется в процессе поиска:

- Оператор нашёл **новое лучшее решение** → большой бонус
- Оператор нашёл **улучшающее решение** → средний бонус
- Оператор нашёл **допустимое решение** → малый бонус
- Оператор не помог → вес снижается

#### Что это даст

- поиск начнет сам понимать, какие операторы работают лучше на твоих данных;
- обычно качество решений заметно растет.

### 5. Делать multi-start и хранить elite pool

Один запуск LNS — это хорошо.  
Но намного лучше:

- несколько сидов;
- несколько стартовых решений;
- хранить pool лучших решений;
- периодически смешивать решения (path relinking / recombination).

### 6. Добавить reheating в simulated annealing

Сейчас температура просто убывает.  
Поэтому иногда поиск "застывает".

#### Что помогает

Если `N` итераций нет улучшений, то чуть поднять температуру снова.

Это называется **reheating**.

#### Эффект

- помогает выйти из плато;
- особенно полезно на длинных запусках.
