# Упрощённый MIP solver для решения задачи первого дня

Программа `mip_solver_day.py` решает **упрощённую задачу маршрутизации героев** для **первого дня недели** с помощью **MIP / MILP**  
(Mixed Integer Linear Programming, смешанное целочисленное линейное программирование).

Программа показывает другой стиль решения задач маршрутизации. Не эвристикой вроде **LNS**, а через **математическую модель**:
- задаём переменные,
- задаём ограничения,
- задаём целевую функцию,
- отдаём задачу MIP-солверу.

## Что делает этот скрипт

1. загружает данные задачи;
2. берёт только **объекты первого дня**;
3. берёт первые `K` героев;
4. строит MIP-модель маршрутов;
5. решает её с помощью **Gurobi**, **HiGHS** или **CBC**;
6. сохраняет:
   - `debug.log`
   - `results.json`
   - `submissions.csv`

## Установка

### Python
Рекомендуется **Python 3.10+**

### Библиотеки Python
Установите:

```bash
pip install numpy pandas pulp highspy
```

Это даст возможность использовать солверы CBC и HiGHS.

## Какие MIP-солверы поддерживаются

Скрипт умеет работать через **PuLP** со следующими солверами:

- **HiGHS**
- **CBC**
- **Gurobi**

### 1. HiGHS

Хороший открытый солвер. Рекомендую начать с него.

Запуск:

```bash
python mip_solver_day1.py --solver highs
```

### 2. CBC
Классический открытый солвер, работает "из коробки" вместе с PuLP.

Запуск:

```bash
python smip_solver_day1.py --solver cbc
```

### 3. Gurobi

Это обычно самый сильный и быстрый вариант. Часто работает в десятки и сотни раз быстрее open-source аналогов.

#### Нужно:

- установленный Gurobi
- доступная лицензия
- чтобы PuLP видел `pulp.GUROBI`

Если всё настроено правильно, можно запускать так:

```bash
python mip_solver_day1.py --solver gurobi
```

## Как запустить

Пример:

```bash
python mip_solver_day1.py \
  --data-dir data \
  --output-dir out \
  --heroes 17 \
  --time-limit 3000 \
  --threads 4 \
  --solver highs
```

## Параметры запуска

- `--data-dir` — папка с входными данными
- `--output-dir` — папка, куда сохраняются результаты
- `--heroes` — сколько первых героев использовать
- `--time-limit` — лимит времени для солвера, в секундах
- `--threads` — число потоков
- `--solver` — какой солвер использовать:
  - `auto`
  - `gurobi`
  - `highs`
  - `cbc`
- `--log-level` — уровень логирования:
  - `DEBUG`
  - `INFO`
  - `WARNING`
  - `ERROR`

После запуска в папке результата будут:

- `debug.log` — подробный лог работы

## Почему MIP / MILP

### Идея

Мы не пишем алгоритм вида:

- "попробовать вставить объект"
- "удалить объект"
- "улучшить маршрут"

Вместо этого мы говорим солверу:

> "Вот все возможные решения в виде переменных.  
> Вот правила задачи в виде линейных ограничений.  
> Найди лучшее."

## Как задача превращается в математическую модель

В данном разделе будет достаточно много формул. Если что-то непонятно, то можно посмотреть на разбор упрощенного примера для 2-х героев и 3-х мельниц [здесь](mip_example.md).

Точная формулировка задачи для первого дня приведена [здесь](mip_math.md).

### Шаг 1. Выбираем переменные

#### 1. Переменная активности героя

Для каждого героя `k` есть бинарная переменная:

$a_k \in \{0,1\}$

Смысл:
- $a_k = 1$, если герой используется;
- $a_k = 0$, если герой не используется.

#### 2. Переменная посещения объекта
Для каждого объекта `i` и героя `k`:

$v_{i,k} \in \{0,1\}$

Смысл:
- $v_{i,k} = 1$, если герой `k` посещает объект `i`;
- $v_{i,k} = 0$, иначе.

Важно: в самой модели таверна не вводится как обычная вершина графа (это отдельная сущность).

Старт из таверны учитывается отдельно через **start cost**.

#### 3. Переменная дуги маршрута

Для каждой пары разных объектов `i, j` и героя `k`:

$x_{i,j,k} \in \{0,1\}$

Смысл:
- $x_{i,j,k} = 1$, если герой `k` идёт по маршруту из `i` в `j`;
- $x_{i,j,k} = 0$, иначе.

То есть именно эти переменные описывают порядок объектов в маршруте.

#### 4. MTZ-переменные порядка

Для каждого объекта `i` и героя `k`:

$u_{i,k} \ge 0$

Это не "реальная" игровая переменная, а **служебная переменная**.  
Она нужна, чтобы запрещать циклы.

Подробнее об MTZ-переменных будет далее.

## Целевая функция

Внутри модели число героев фиксировано заранее, поэтому их стоимость — константа и на оптимум не влияет.

Значит в модели мы максимизируем:

$\sum_{i,k} v_{i,k}$

То есть просто **число посещённых объектов**.

### Tie-breaker по leftover

Чтобы среди решений с одинаковым числом объектов выбирать более "аккуратные", добавлен маленький коэффициент:

$$
\max \left(
\sum_{i,k} v_{i,k}
+ \varepsilon \cdot \text{leftover}
\right)
$$

где `epsilon` очень маленький (например, 0.000001).

#### Зачем это нужно

Если два решения посещают одинаковое число объектов, лучше выбрать то, где:
- герои потратили меньше хода,
- осталось больше запаса.

Такой tie-breaker похож на то, что часто делают в эвристиках.

## Основные ограничения модели

### 1. Каждый объект можно посетить не более одного раза

$\sum_k v_{i,k} \le 1$

Это означает:
- объект либо не посещён,
- либо посещён ровно одним героем.

### 2. Если герой не активен, он не может посещать объекты

$\sum_i v_{i,k} \le n \cdot a_k$

где `n` — число объектов дня.

Если $a_k = 0$, правая часть равна 0, значит герой не может посетить ни одного объекта.

### 3. У активного героя должен быть ровно один старт и ровно один конец

#### Старт

Стартом считается объект, который посещён, но в который не входит ни одна дуга.

Для объекта `i`:
- посещён: $v_{i,k}$
- входящих дуг: $\sum_p x_{p,i,k}$

Значит "быть стартом" можно записать как:

$v_{i,k} - \sum_p x_{p,i,k}$

Если сумма по всем объектам равна 1, значит старт ровно один:

$\sum_i \left(v_{i,k} - \sum_p x_{p,i,k}\right) = a_k$

#### Конец
Аналогично, конец — это объект, который посещён, но из которого не выходит дуга:

$\sum_i \left(v_{i,k} - \sum_s x_{i,s,k}\right) = a_k$

### 4. Если объект посещён, вход и выход через него может быть только один

$\sum_p x_{p,i,k} \le v_{i,k}$

$\sum_s x_{i,s,k} \le v_{i,k}$

Это также значит: если объект не посещается, через него нельзя провести дуги.

### 5. Запрет недопустимого старта

Первый объект маршрута должен быть достижим из таверны.

Если для героя `k` объект `i` не может быть стартом, то:

$v_{i,k} \le \sum_p x_{p,i,k}$

То есть если герой посетил такой объект, у него обязательно должен быть входящий переход.  
Значит он не может стоять первым.

## Ограничение по очкам хода

Это одна из самых интересных частей модели.

## Стоимость маршрута состоит из:

1. дороги от таверны до первого объекта;
2. переходов между объектами;
3. стоимости посещений.

### Формула
Для героя `k`:

$$
\text{used}_k
=
\text{start\_cost}_k
+
\text{arc\_cost}_k
+
100 \cdot \text{visit\_count}_k
-
99 \cdot a_k
$$

И затем:

$\text{used}_k \le \text{capacity}_k$

### Как здесь учитывается правило последнего хода

В игре последнее посещение можно сделать, если у героя осталось хотя бы 1 очко хода.

#### Без last move
Если бы last move не было, каждое посещение стоило бы `100`.

#### С last move
Последнее посещение можно считать как будто стоит `1` вместо `100`.

Это и делается через поправку:

$-99 \cdot a_k$

Почему это работает:
- у активного героя ровно один маршрут;
- значит у него ровно одно "последнее посещение";
- именно для него `100` превращается в `1`.

## Зачем нужны MTZ-переменные

Это одна из самых важных частей модели.

### Проблема
Даже если мы правильно задали:
- посещения;
- входы;
- выходы;

солвер всё ещё может построить **подциклы**.

Например:
- один кусок маршрута идёт нормально,
- а ещё отдельно существует цикл `A -> B -> C -> A`.

Такое решение формально может удовлетворять многим ограничениям, но это не один нормальный маршрут.

### Идея MTZ
MTZ-переменные $u_{i,k}$ задают **порядок объекта в маршруте**.

Например:
- первый объект имеет порядок 1,
- второй — 2,
- третий — 3,
- и так далее.

Если дуга `i -> j` используется, то порядок должен увеличиваться:

$u_{j,k} \ge u_{i,k} + 1$

В линейной форме это записывается так:

$u_{i,k} - u_{j,k} + n \cdot x_{i,j,k} \le n - 1$

где `n` — число объектов.

### Почему это запрещает цикл

Представим цикл:

$A \to B \to C \to A$

Тогда ограничения требуют:

- $u_B \ge u_A + 1$
- $u_C \ge u_B + 1$
- $u_A \ge u_C + 1$

Сложив их, получим невозможность:

$u_A + u_B + u_C \ge u_A + u_B + u_C + 3$

Значит цикл запрещён.

MTZ — это хороший пример того, как: сложное логическое правило, вроде "не должно быть цикла", можно превратить в **обычные линейные ограничения**.

Это одна из самых красивых идей в классическом MILP для маршрутизации.

## Пояснения по модели

### Почему в формулах пишется $d_{12}$ и $d_{21}$, если расстояние одно и то же?

#### Короткий ответ

Потому что модель записывается **по дугам**:

$x_{i,j,k}$

То есть мы описываем переход **из $i$ в $j$**.  
Поэтому и расстояние удобно обозначать так же:

$d_{i,j}$

#### Важно

Если матрица расстояний симметрична, то просто:

$d_{12} = d_{21}$

То есть запись с двумя индексами не означает, что расстояния обязательно разные.  
Это просто **более общий стиль записи**.

#### Почему так удобно

Такой стиль:
- подходит и для симметричных задач;
- и для несимметричных;
- и сразу согласован с направленными переменными $x_{i,j,k}$.

### Почему MTZ-переменные $u_{i,k}$ непрерывные, а не целые?

На словах кажется, что $u_{i,k}$ — это "номер объекта в маршруте",  
значит он должен быть целым.

- 1-й объект,
- 2-й объект,
- 3-й объект,
- ...

Но в модели это **служебные переменные**, а не "настоящие номера шага".

#### Их задача

Они нужны только для одного: **запрещать подциклы**

#### Почему целочисленность не нужна

Если есть цикл, MTZ-ограничения делают систему противоречивой
**и для целых, и для вещественных значений**.

То есть для запрета циклов достаточно, чтобы $u$ были просто непрерывными.

#### Зачем это полезно

Если сделать $u$ целыми:
- модель станет тяжелее;
- солвер будет работать медленнее.

Если сделать $u$ непрерывными:
- модель проще;
- решение обычно быстрее;
- смысл ограничений сохраняется.

Поэтому в классических MTZ-постановках $u$ часто делают именно **непрерывными**.

### Что такое leftover?

Для героя `k`:

$\text{leftover}_k = \text{capacity}_k \cdot a_k - \text{used}_k$

Смысл:
- сколько хода осталось у героя после выполнения маршрута.

Если герой не активен:
- $a_k = 0$
- leftover получается равным 0.

## Как извлекается маршрут из решения

После решения модели мы знаем:

- какие объекты посещены (`v`)
- какие дуги выбраны (`x`)

Дальше:

1. находим все посещённые объекты;
2. для каждого считаем входящую степень;
3. старт — это объект с входящей степенью 0;
4. потом идём по дугам `x`, пока маршрут не закончится.

Так из бинарных переменных получается обычный список объектов:
```text
hero 1: 17 -> 33 -> 45
hero 2: 8 -> 12
...
```

## Преимущества MILP-подхода по сравнению с LNS

### 1. Очень чёткая математическая постановка

В LNS вы пишете:
- как разрушать решение,
- как чинить,
- какие эвристики использовать.

В MILP вы пишете:
- что такое допустимое решение;
- что именно нужно максимизировать.

Это часто проще для формального анализа.

### 2. Можно доказывать оптимальность

Если солвер вернул статус `Optimal`, это значит:

> лучшее решение найдено и доказано математически.

Для LNS такого обычно нет:
- LNS может найти очень хорошее решение,
- но не может доказать, что оно лучшее возможное.

### 3. Меньше "магии" в логике алгоритма

В LNS много эвристических решений:
- какой destroy выбрать,
- какой repair выбрать,
- сколько удалять,
- какие параметры температуры,
- какие случайные seeds запускать.

В MILP всё это не нужно.  
Есть модель и солвер.

## Недостатки MILP по сравнению с LNS

### 1. Часто медленнее на больших задачах

MILP может быть очень тяжёлым:
- много переменных,
- много ограничений,
- долгое время решения.

Например, для 17 героев получается более 300000 переменных и более 300000 ограничений. Время поиска решения может занять много дней.

### 2. Требует математической модели

Чтобы написать MILP, нужно:
- аккуратно придумать переменные;
- аккуратно придумать ограничения.

Это сложнее, чем написать простую эвристику.

### 3. Меньше гибкости

Если в задаче очень много игровых тонкостей, иногда проще встроить их в эвристику, чем в строгую линейную модель.

## Потенциальные улучшения базовой MIP-версии

Ниже перечислены идеи, как эту простую модель можно сделать сильнее и быстрее.

### 1. Учитывать симметрию героев

Во входных данных часто бывает так, что несколько героев имеют одинаковый `move_points`.

Это значит, что они **симметричны**: если поменять их маршруты местами, решение по сути не изменится.

#### Почему это плохо

С точки зрения солвера это создаёт много одинаковых по смыслу решений.  
Солвер тратит время на перебор "одного и того же".

#### Что можно сделать

Добавить **symmetry breaking constraints**, например:
- требовать, чтобы у героя с меньшим индексом маршрут был не "хуже", чем у следующего героя той же capacity;
- упорядочить число посещённых объектов между симметричными героями;
- упорядочить стартовые вершины.

Это обычно заметно ускоряет поиск.

### 2. Добавить clique cuts

Если заранее выяснить, что некоторые пары объектов нельзя посетить одним и тем же героем ни в каком порядке, можно строить **граф несовместимости**.

Тогда из этого графа можно получить клики и добавить ограничения вида:

$\sum_{i \in clique} v_{i,k} \le 1$

#### Зачем это нужно

Такие ограничения заранее вырезают много невозможных комбинаций объектов и уменьшают пространство поиска.

### 3. Добавить warm start

Сейчас модель стартует "с нуля".

#### Что можно сделать

Перед запуском солвера:
- построить жадное решение;
- или взять маршрут из LNS;
- или взять решение из предыдущего запуска.

Потом передать его солверу как начальное решение.

#### Зачем

Хороший warm start часто:
- быстро даёт хороший базовый вариант;
- улучшает ветвление;
- помогает доказать оптимальность.

### 4. Пробовать другую формулировку модели

Самый простой вариант, это сделать таверну и мельницы равнозначными. Это упростит часть формул.

Или можно ввести явные переменные старта и конца маршрута.

#### Почему это интересно

Иногда одна и та же задача в другой системе переменных решается гораздо быстрее.

### 5. Route-based модель вместо arc-based

Сейчас модель выбирает:
- посещения объектов,
- дуги между объектами.

А можно делать иначе:

1. заранее сгенерировать набор допустимых маршрутов;
2. в MIP выбирать не дуги, а целые маршруты.

Тогда переменная будет такая: "выбрать маршрут r для героя k или нет".

Я пытался реализовать такой вариант, но число маршрутов для первого дня получается слишком большим (точно больше миллиарда). Но возможно получится применить этот вариант для следующих дней или для каких то подмножеств мельниц.

### 6. Убрать `epsilon` и перейти на двухфазную оптимизацию

Сейчас целевая функция такая:

$$
\text{visited} + \varepsilon \cdot \text{leftover}
$$

Это нормально, но у таких конструкций есть два минуса:

- возможны численные эффекты;
- некоторые солверы любят такие масштабы меньше, чем явную лексикографию.

#### Что можно сделать

Сделать **две фазы**:

##### Фаза 1

Максимизировать только число посещений:

$$
\max \sum v_{i,k}
$$

##### Фаза 2

Добавить ограничение:

$$
\sum v_{i,k} = \text{opt\_visited}
$$

и потом максимизировать leftover.

# Итог

Эта программа показывает, как задачу маршрутизации можно решать не эвристикой, а через **математическую оптимизацию**:

- вводим переменные;
- записываем правила в виде линейных ограничений;
- запрещаем циклы через MTZ;
- максимизируем число посещённых объектов.
